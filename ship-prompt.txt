EXECUTION — Fix one ticket at a time:

2) CHECK CONVERGENCE: has this file been fixed 3+ times? If so, DO NOT FIX.
   Log as ESCALATE in PROGRESS.md. Delete ticket. Move on.
3) READ the broken code. Understand the root cause.
4) FIX it. Minimal change. Follow DESIGN.md.
5) VERIFY:
   a) Re-read the code. Does the fix meet acceptance criteria?
   b) RUN SMOKE TESTS AFTER EVERY FIX:
      python functional-qa/tests/smoke.py
      If any previously-passing test now FAILS → REVERT your changes:
      git checkout -- [files you changed]
      Log: "Fix for [ticket] reverted — broke [test name]"
      This is the autoresearch keep/discard pattern. Metric better → keep. Worse → revert.
   c) If frontend was touched, VISUALLY VERIFY:
      node functional-qa/browse.js diff "lab.html"
      Or take a targeted screenshot and Read it. Max 3 screenshots per fix.
      Vision is for layout, not data.
6) DELETE the ticket file: rm tickets/[filename].md
7) LOG the fix in PROGRESS.md (include ticket ID so QA Loop re-audits)
8) COMMIT and PUSH:
   git add [fixed files] PROGRESS.md
   git rm tickets/[filename].md
   git commit -m "Fix: [ticket description] (ticket [ID])"
   git push origin ship/launch-fixes
9) NEXT TICKET. Do not stop. Do not summarize. Keep fixing.

CONFIDENCE RULES:
- HIGH: QA is certain. Trust it. Auto-fix.
- MEDIUM: QA thinks it's a bug. Fix, but verify extra carefully.
- LOW: QA isn't sure. DO NOT FIX. Leave the ticket file. Human decides.
- ESCALATE: Fix-break cycle. DO NOT FIX. Log in PROGRESS.md. Delete ticket. Move on.

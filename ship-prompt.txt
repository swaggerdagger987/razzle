EXECUTION — Fix one ticket at a time:

1) READ the ticket file. It has file:line:function and acceptance criteria.
2) CHECK CONVERGENCE: has this file been fixed before? Search PROGRESS.md and tickets/.
   If 3+ prior attempts on the same file, DO NOT FIX. Log as ESCALATE in PROGRESS.md. Delete ticket. Move on.
3) READ the broken code. Understand the root cause.
4) FIX it. Minimal change. Follow DESIGN.md.
5) VERIFY — three checks:
   a) Re-read the code. Does the fix meet acceptance criteria?
   b) If backend was touched: python -m pytest tests/ -v
   c) If frontend was touched: VISUALLY VERIFY your fix:
      node functional-qa/browse.js diff "lab.html"   (compare against baseline)
      — or —
      node functional-qa/browse.js script '[{"action":"load","url":"lab.html"},{"action":"wait","ms":3000},{"action":"screenshot","filename":"fix-verify.png"}]'
      Then: Read functional-qa/screenshots/fix-verify.png to SEE your fix
      Vision is for layout, not data. Don't try to read numbers from screenshots.
      Max 3 screenshots per fix. Be strategic.
6) DELETE the ticket file: rm tickets/[filename].md
7) LOG the fix in PROGRESS.md (include ticket ID so QA Loop re-audits)
8) COMMIT and PUSH:
   git add [fixed files] PROGRESS.md
   git rm tickets/[filename].md (if not already deleted)
   git commit -m "Fix: [ticket description] (ticket [ID])"
   git push origin ship/launch-fixes
9) NEXT TICKET. Do not stop. Do not summarize. Keep fixing.

CONFIDENCE RULES:
- HIGH: The QA Loop is certain. Trust it. Auto-fix.
- MEDIUM: The QA Loop thinks it's a bug. Fix it, but verify extra carefully.
- LOW: The QA Loop isn't sure. DO NOT FIX. Leave the ticket file. Human decides.
- ESCALATE: Fix-break cycle detected. DO NOT FIX. Log in PROGRESS.md. Delete ticket. Move on.

SWEEP MODE (only when tickets/ AND TICKETS.md are empty):

- Fix ONE issue per page (design, functionality, UX, brand voice)
- Commit and push each fix
- Check tickets/ every few fixes — new QA tickets interrupt sweep

# Razzle Review -- 2026-03-14
# FOCUS: Owner/Admin Access, Tier Splitting Edge Cases, Production Readiness

## Ship Readiness: 52/100

## Executive Summary

Razzle has made significant progress since the March 13 review (34/100 to 52/100) -- the build pipeline, mobile nav, CSS variable audit, security headers, BYOK cleanup, and season defaults are all fixed. However, this focused audit reveals three structural gaps that will cause real problems at launch: (1) **There is zero admin functionality** -- no admin role, no admin panel, no way to see who's paying, no way to grant access to press/reviewers, no way to manually upgrade/downgrade users without direct SQLite access; (2) **Tier enforcement is split-brain** -- 9 distinct features are gated client-side only (via localStorage manipulation), with the server enforcing tiers on only 5 categories of endpoints; (3) **Stripe webhook coverage is dangerously incomplete** -- only 3 of ~12 critical events are handled, meaning subscription updates, trial expirations, plan changes, and disputes all go unprocessed. Any real user who modifies their subscription through Stripe's customer portal will silently desync from Razzle's database.

## SPECIFIC ANSWERS TO OWNER QUESTIONS

### 1. Is there ANY admin role or admin access?

**NO. There is zero admin functionality anywhere in the codebase.**

- The `users` table has no `role` or `is_admin` column (auth.py:93-101)
- No admin middleware, no admin routes, no admin panel exists
- The health endpoint (`/api/health`) shows diagnostic info to any authenticated user, but has no admin restriction (server.py:707-715)
- There is no way to:
  - See a list of registered users
  - See who is paying vs free vs trial
  - View subscription statuses
  - See revenue metrics
  - Handle support requests
  - Ban or suspend users
  - View error logs from a browser

**How do you currently manage users?** You cannot. You would need to SSH into the Render instance and run SQLite queries directly against `/data/users.db`. There is no CLI tool, no management script, no dashboard.

**Severity: CRITICAL**

### 2. Can the owner bypass payment to access all features?

**Not through the application.** There is no "superuser" or "owner" flag. The only ways to grant yourself full access are:

1. **Direct database manipulation**: SSH into Render, open users.db, run `UPDATE users SET plan = 'elite_lifetime' WHERE email = 'your@email.com'`
2. **Forge a JWT**: Since you control JWT_SECRET, you could manually craft a token with `plan: "elite"`, but there's no tool or endpoint to do this
3. **Fake a Stripe webhook**: POST to `/api/billing/webhook` with a crafted checkout.session.completed event, but you'd need the webhook signature

None of these are practical workflows for day-to-day access. There is no `/api/admin/grant-access` or similar.

**Severity: HIGH** -- You need at minimum a way to grant yourself and press/reviewers access.

### 5. Tier transition edge cases

| Transition | What happens | What SHOULD happen | Gap |
|------------|-------------|-------------------|-----|
| Register -> Trial | Auto 7-day Pro trial starts, token carries `plan: "pro"` | Correct | None |
| Trial -> Expired | `_user_dict()` checks `trial_end > now`, returns `plan: "free"` | Correct | **But** the JWT token still carries `plan: "pro"` until it expires (7 days). A user whose trial expired could still have a valid JWT claiming Pro for up to 7 days. |
| Pro -> Cancelled | `customer.subscription.deleted` webhook sets `users.plan = 'free'` | Correct | **But** existing JWT still carries old plan until expiry. And `customer.subscription.updated` is not handled, so if Stripe cancels at period end (the normal flow), nothing happens until `deleted` fires. |
| Elite -> Pro downgrade | Not handled | Should update `users.plan` to `pro` | **MISSING**: `customer.subscription.updated` webhook not handled. Stripe plan changes are invisible to Razzle. |
| Pro -> Elite upgrade | Handled via new checkout session | Should work | Untested path -- does creating a new subscription while one exists cause double-billing? No idempotency check. |
| Payment failed | `invoice.payment_failed` sets `subscriptions.status = 'payment_failed'` but does NOT change `users.plan` | Should downgrade after grace period | **User keeps full paid access even after payment failure.** |
| Lifetime -> anything | Protected -- `_handle_subscription_deleted` skips lifetime plans | Correct | None |
| Free -> Free (re-register for new trial) | `trial_used = 1` on users table prevents re-trial | Correct for CC trial | **But** a user could create a new account with a different email and get another free trial. No device fingerprinting or other mitigation. |

### 6. Can localStorage manipulation bypass ANY tier gate?

**YES. Extensively.**

Setting `localStorage.setItem("razzle_user", JSON.stringify({plan: "elite", trial_active: false, plan_source: "subscription"}))` would immediately unlock:

1. All seasons in the season selector (app.js:244)
2. Unlimited formulas (app.js:257)
3. 4-player compare mode (app.js:259, lab.js:4452)
4. CSV export (lab.js:8)
5. Full Formula Store access (formula-store.js:470)
6. Full manager profiles (5yr history) in Bureau (league-intel.html:1581)
7. Full pressure map (league-intel.html:2210)
8. BYOK AI queries -- the frontend would send queries to OpenRouter with a user's own key, bypassing all server-side gating (warroom.js)
9. League context mode in AI (warroom.js:1900)
10. Hide AdSense ads (app.js:1263)
11. Activity feed paid features (league-intel.html:1402)

**What it would NOT bypass:**
- Server-side LLM proxy (require_plan "elite" on server.py:1001)
- Agent memory cloud sync (require_plan "elite" on server.py:1307)
- Watchlist/views cloud sync (require_plan "pro" on server.py:1441-1480)
- BYOK key cloud storage (require_plan "pro" on server.py:1210)
- League trade finder API (require_plan "pro" on server.py:2620)
- Weekly briefings (require_plan "elite" on server.py:1359)

**Summary**: A user with browser DevTools could get most of the Pro experience for free. The things they can't bypass are cloud sync features and the server-side LLM proxy. The most damaging bypass is BYOK AI -- a free user who sets their localStorage plan to "pro" and enters their own OpenRouter key could use the Situation Room with full league context without paying, because all the LLM calls go directly from the browser to OpenRouter (not through Razzle's server).

**Severity: HIGH** -- Not a security catastrophe (no data leaks, no server compromise), but it means anyone who reads the source code can get Pro-tier features for free. For a product targeting Reddit power users (who will absolutely inspect source), this is a real revenue risk.

### 7. Is there a way to grant someone access manually (press/reviewers)?

**NO.** There is no:
- Admin panel to upgrade users
- Admin API endpoint
- Promo code that grants free access (the RAZZLE promo code maps to a Stripe coupon for a discount, not free access)
- Invite/referral system
- Comped account mechanism

The only option is direct SQLite manipulation on the production server.

**Severity: HIGH** -- You will need to give access to reviewers, friends, beta testers. Right now you cannot do this without SSH.

### 8. Stripe webhook events: handled vs missing

**Handled (3):**
| Event | Handler | What it does |
|-------|---------|-------------|
| `checkout.session.completed` | `_handle_checkout_completed()` | Sets `users.plan`, creates subscription record |
| `customer.subscription.deleted` | `_handle_subscription_deleted()` | Downgrades to free (skips lifetime) |
| `invoice.payment_failed` | `_handle_payment_failed()` | Sets subscription status to `payment_failed` (but does NOT change user plan) |

**Missing (critical):**
| Event | Why it matters | Risk |
|-------|---------------|------|
| `customer.subscription.updated` | Plan changes (upgrade/downgrade), cancellation scheduling, quantity changes | **User's plan in Razzle will desync from Stripe.** If someone downgrades from Elite to Pro in Stripe's portal, Razzle will still show Elite. |
| `customer.subscription.paused` | Stripe supports pausing subscriptions | User keeps access while paused |
| `customer.subscription.resumed` | Resume after pause | Razzle won't know to re-enable |
| `invoice.paid` | Confirms successful payment after retry | `payment_failed` status never clears |
| `invoice.payment_action_required` | 3D Secure / SCA authentication required | Payment stuck in limbo |
| `customer.subscription.trial_will_end` | 3 days before trial ends | No way to send "trial ending" email |
| `charge.dispute.created` | Chargeback filed | User keeps access during dispute |
| `charge.refunded` | Full/partial refund issued | User keeps access after refund |
| `checkout.session.expired` | User abandoned checkout | No cleanup of pending state |

**Severity: CRITICAL** -- The `customer.subscription.updated` omission alone means any plan change through Stripe's billing portal will silently desync. Since the pricing page links to Stripe's portal for subscription management (`billing_portal.Session.create` in billing.py:504), this IS the expected path for changes.

## Additional Critical Findings

### JWT Plan Caching Bug
**Severity: CRITICAL**
**File: backend/auth.py:223-231, 276-278, 330-331**

The JWT token includes the user's plan at the time of login/registration. During a trial, the token is minted with `plan: "pro"`. However, `get_current_user()` (auth.py:355-368) fetches the fresh user record from the database and returns the current effective plan via `_user_dict()`. This means the JWT plan and the database plan can diverge.

The real problem: `require_plan()` in server.py:749-760 calls `require_auth()` which calls `get_current_user()` which reads from the database. So server-side enforcement IS based on the fresh DB state. Good.

But the frontend stores `razzle_user` in localStorage at login time and only refreshes it on page load via `/api/auth/me` (app.js:706-727). If a user's trial expires while they have the page open, they keep Pro access until they refresh. This is a minor issue.

**However**: The `/api/auth/me` refresh (app.js:706) runs on every page load. If it fails (network error, server down), the code falls back to the cached localStorage user (app.js:723). This means a stale plan could persist across page loads during server outages.

### Payment Failure Does Not Revoke Access
**Severity: CRITICAL**
**File: backend/billing.py:451-463**

When `invoice.payment_failed` fires, the handler updates `subscriptions.status` to `payment_failed` but does NOT change `users.plan`. The user continues to have full paid access indefinitely. There is no grace period logic, no retry tracking, no escalation to downgrade.

This means: a user whose credit card declines will keep full Pro/Elite access forever (or until Stripe eventually fires `customer.subscription.deleted` after its own retry cycle, typically 3-4 weeks).

### No Password Reset
**Severity: HIGH**
**File: backend/auth.py (entire file), backend/server.py (all auth endpoints)**

There is no password reset flow. No `/api/auth/forgot-password`, no email sending capability, no reset token mechanism. A user who forgets their password is permanently locked out of their account. They would need to contact support (which also doesn't exist as a feature) or create a new account (losing their data).

### Formula Cloud Sync Has No Plan Check
**Severity: MEDIUM**
**File: backend/server.py:1258-1297**

The formula CRUD endpoints (`/api/user/formulas`, POST, DELETE, import) use `require_auth()` only -- no `require_plan()` check. Any registered user (including free) can sync formulas to the cloud. This contradicts the pricing page which lists "Formula cloud sync" as a Pro feature.

Compare with watchlist sync (server.py:1465) and saved views sync (server.py:1441) which correctly use `require_plan("pro")`.

### Early Adopter Counting Bug
**Severity: MEDIUM**
**File: backend/billing.py:119-124**

`get_subscriber_counts()` counts users with `plan IN ('pro', 'pro_lifetime')` for Pro slots and `plan IN ('elite', 'elite_lifetime')` for Elite slots. But trial users have `users.plan = 'free'` (their effective plan is computed at read time). And early adopter users get `plan = 'pro'` (same as regular Pro). So the early adopter cap counter includes ALL Pro subscribers, not just early adopter ones. If you have 500 regular Pro users, no one can get the early adopter deal even if zero early adopters exist.

### No Email Verification
**Severity: MEDIUM**
**File: backend/auth.py:298-335**

Users can register with any email address without verification. Combined with the 7-day free trial, this means:
- Someone can create unlimited trial accounts with fake emails
- No way to contact users for support, announcements, or trial expiry warnings
- Registration rate limit (3 per IP per 24h) is the only mitigation, easily bypassed with a VPN

### Elite Users Get Free Model Instead of Premium
**Severity: MEDIUM**
**File: backend/server.py:1040-1045**

The LLM proxy for Elite users uses `_LLM_FREE_MODEL` (llama-3.1-8b-instruct:free) instead of `_LLM_MODEL` (claude-3.5-haiku). The comment says "use free model -- zero marginal cost" but this means Elite users paying $149.99/yr get the same free model quality as free-tier users. The pricing page promises "No API key needed -- we pay for it" and "Unlimited AI queries (included)" which implies premium model access.

### No Subscription Idempotency
**Severity: MEDIUM**
**File: backend/billing.py:278-319, server.py:847-865**

There is no check for whether a user already has an active subscription before creating a new checkout session. A Pro user clicking "Get Elite" will create a second subscription without cancelling the first, resulting in double billing. Stripe handles this somewhat through their customer portal, but the Razzle code does not.

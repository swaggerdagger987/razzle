# Razzle Accessibility Audit -- 2026-03-14

## Persona: Accessibility Auditor
## Standard: WCAG 2.1 Level AA
## Scope: All 74 HTML files, styles.css, lab-panels.css, app.js, lab.js, warroom.js

---

## Executive Summary

Significant accessibility improvements since the 2026-03-13 review. The prior review's CRITICAL items -- `maximum-scale=1.0` on all 74 pages, missing `role="dialog"` on modals, missing nav `aria-label`, and non-focusable search hint -- have ALL been fixed. The auth modal now has focus trapping, Escape key handling, and return-focus behavior. Skip-to-content links exist on all 74 pages. The `<nav>` element has `aria-label="Main navigation"` on all 74 pages. `scope="col"` is used on the screener table headers. The hamburger menu has `aria-expanded` state management. The canvas in agents.html has `role="img"` and a descriptive `aria-label`. `prefers-reduced-motion` is respected in styles.css.

However, several MEDIUM and HIGH issues remain. The codebase does not conform to WCAG 2.1 Level AA, though it is much closer than before.

### 7. [HIGH] Auth form inputs lack visible labels -- WCAG 1.3.1, 3.3.2
- **File**: `frontend/app.js:541-549`
- **Evidence**: Auth form inputs use `placeholder` as the only label:
  ```
  '<input type="email" id="authLoginEmail" placeholder="Email" required autocomplete="email">'
  '<input type="password" id="authLoginPassword" placeholder="Password" required autocomplete="current-password">'
  ```
- **Problem**: Placeholder text disappears when the user starts typing, leaving no persistent label. Screen readers may not consistently announce placeholder as a label (behavior varies by AT). No `<label>` elements or `aria-label` attributes are used.
- **WCAG**: 1.3.1 (Info and Relationships), 3.3.2 (Labels or Instructions)
- **Recommendation**: Add `aria-label` attributes to each input (e.g., `aria-label="Email address"`), or add visually hidden `<label>` elements with `for` attributes. Better yet, use visible labels.

### 8. [HIGH] outline:none without focus-visible replacement on many elements -- WCAG 2.4.7
- **Files**: Multiple locations
  - `frontend/styles.css:603` -- `.auth-form input` removes outline. Has `:focus` replacement (border-color + box-shadow) AND `:focus-visible` (line 611) -- this one is actually OK.
  - `frontend/styles.css:783` -- `.input-chunky` removes outline. Has `:focus-visible` (line 790) -- OK.
  - `frontend/styles.css:811` -- `.select-chunky` removes outline. Has `:focus-visible` (line 813) -- OK.
  - `frontend/styles.css:950` -- `.cmd-palette-input` removes outline. Has `:focus-visible` (line 956) -- OK.
  - `frontend/styles.css:1334` -- `.note-editor-input:focus` removes outline. No `:focus-visible` replacement. **VIOLATION.**
  - `frontend/agents.html:337` -- `.scenario-textarea:focus` removes outline. Has border-color + box-shadow replacement but no `:focus-visible`. **VIOLATION** (border-color change alone may not be sufficient for all users).
  - `frontend/agents.html:1384` -- `.config-input:focus` removes outline. **VIOLATION.**
  - `frontend/agents.html:1459` -- `.config-agent-key:focus` removes outline. **VIOLATION.**
  - `frontend/breakdown.html:66` -- `.bd-search:focus` removes outline. **VIOLATION.**
  - `frontend/lab.html:133` -- `.sidebar-search` removes outline. Has `:focus` border-color change but no `:focus-visible` with outline. **VIOLATION.**
  - `frontend/rosterbuilder.html:63`, `frontend/strengths.html:67`, `frontend/tradefinder.html:74`, `frontend/tools.html:78` -- All remove outline without `:focus-visible` replacement.
  - `frontend/lab-panels.css:4300` -- removes outline. **VIOLATION.**
- **WCAG**: 2.4.7 (Focus Visible)
- **Recommendation**: For every `outline: none`, add a `:focus-visible` rule that provides a visible 2px+ outline or equivalent indicator. The pattern used in styles.css (`.input-chunky:focus-visible { outline: 3px solid var(--orange); outline-offset: 2px; }`) is correct -- replicate it everywhere.

### 9. [HIGH] Lab dialogs lack aria-labelledby -- WCAG 4.1.2
- **File**: `frontend/lab.html:3272-3764`
- **Evidence**: All 19 dialog overlays have `role="dialog"` and `aria-modal="true"` but none have `aria-labelledby` or `aria-label`. Each dialog has an `<h3>` inside but it is not referenced.
  ```html
  <div role="dialog" aria-modal="true" class="filter-modal-overlay" id="filterModalOverlay">
    <div class="filter-modal">
      <h3>Add Filter</h3>
  ```
- **Problem**: Screen readers announce these as unlabeled dialogs. Users cannot tell which dialog they are in.
- **WCAG**: 4.1.2 (Name, Role, Value)
- **Recommendation**: Add `id` to each `<h3>` (e.g., `id="filter-modal-title"`) and `aria-labelledby` on each dialog overlay.

### 10. [HIGH] Lab dialogs have no focus trap -- WCAG 2.4.3
- **File**: `frontend/lab.html`, `frontend/lab.js`
- **Evidence**: The auth modal (app.js:614-629) has a focus trap, but the 19 lab-specific modals (filter, column picker, formula, chart, compare, share, saved views, custom scoring, rankings export, tier view, big board, class analytics, trade analyzer, trade values, profile, publish, store, aging curves, heat map) do not have focus trapping code.
- **Problem**: When a dialog is open, Tab can escape to background content. Screen reader users can navigate outside the dialog.
- **WCAG**: 2.4.3 (Focus Order)
- **Recommendation**: Implement a shared focus trap utility and apply it to all modals when opened.

### 11. [HIGH] Dynamic content updates lack aria-live regions -- WCAG 4.1.3
- **Files**: Most panel pages (all 60+ panel HTML files)
- **Evidence**: Only 2 pages use `aria-live`:
  - `frontend/agents.html:1805` -- `aria-live="polite"` on briefing cards (correct)
  - `frontend/index.html:1253` -- `aria-live="polite"` on demo cards (correct)
- **Problem**: When data loads/filters/sorts on panel pages, screen readers are not notified. The screener table in lab.html updates its entire tbody dynamically. Panel pages fetch data and inject HTML. None of these use `aria-live` or any notification mechanism.
- **WCAG**: 4.1.3 (Status Messages)
- **Recommendation**: Add `aria-live="polite"` to the main data container on each panel page. For the screener, add a visually hidden status region that announces "X players found" or "sorting by Y" after updates.

### 12. [MEDIUM] No `<main>` landmark on any page -- WCAG 1.3.1
- **Files**: All 74 HTML pages
- **Evidence**: Pages use `id="main-content"` as the skip-link target but no `<main>` element or `role="main"` is used. Content is in `<div>` containers.
- **Problem**: Screen reader users cannot quickly navigate to the main content area via landmark navigation. The skip link helps, but landmarks are a separate navigation mechanism.
- **WCAG**: 1.3.1 (Info and Relationships) -- best practice for landmarks
- **Recommendation**: Wrap the primary content `<div>` with `<main id="main-content">` instead of `<div id="main-content">`.

### 13. [MEDIUM] No `<h1>` element on panel pages loaded in iframe -- WCAG 1.3.1, 2.4.6
- **Files**: ~60 panel pages (targets.html, aging.html, etc.)
- **Evidence**: Panel pages like `targets.html` are loaded inside lab.html via iframe. They have `<h1>` in their standalone view but when embedded, the heading hierarchy may not align with the parent page. Additionally, `targets.html:7` has a redirect script: `if(window.self===window.top) window.location.replace('/lab.html?panel=targets');` -- meaning the page only loads standalone briefly.
- **Problem**: When loaded in iframe, the page has its own document with its own heading hierarchy. This is acceptable for iframe architecture, but the parent lab.html itself has no visible `<h1>`.
- **WCAG**: 2.4.6 (Headings and Labels)
- **Recommendation**: Ensure lab.html has a visible or sr-only `<h1>` like "The Lab -- Razzle".

### 14. [MEDIUM] Color contrast: --ink-light on --bg sand -- WCAG 1.4.3
- **Values**: `--ink-light: #6d5c4e` on `--bg: #ede0cf`
- **Calculated contrast ratio**: approximately 3.3:1
- **Required**: 4.5:1 for normal text (Level AA)
- **Usage**: Used extensively for labels, metadata, timestamps, and subtitle text across all pages. The prior review noted this at `--ink-light: #8a7565` (2.9:1 ratio). The current value `#6d5c4e` is darker and better but still fails AA for normal-sized text.
- **Note**: At larger sizes (18px+ bold or 24px+), 3:1 is acceptable. Some uses (e.g., Caveat 20px) may pass. But many uses are at 11-13px where 4.5:1 is required.
- **WCAG**: 1.4.3 (Contrast Minimum)
- **Recommendation**: Darken `--ink-light` to at least `#5c4a3d` (which is the current `--ink-medium` value) for small text uses, or create a separate token for small labels that meets 4.5:1.

### 15. [MEDIUM] Color contrast: --ink-faint on --bg -- WCAG 1.4.3
- **Values**: `--ink-faint: #c4b5a5` on `--bg: #ede0cf`
- **Calculated contrast ratio**: approximately 1.5:1
- **Usage**: Used for dividers and dashed borders -- acceptable as decorative. But also used for text in some contexts (tag-icon color, note-pencil).
- **WCAG**: 1.4.3
- **Recommendation**: Ensure `--ink-faint` is never used for text that conveys information. Use it only for decorative borders.

### 16. [MEDIUM] Data tables on panel pages lack semantic structure -- WCAG 1.3.1
- **Files**: Most of the 60+ panel pages (drops.html, dualthreat.html, gamescript.html, garbagetime.html, etc.)
- **Evidence**: Tables are generated via JavaScript with `<thead>` and `<th>` but no `scope="col"` or `scope="row"` attributes. No `<caption>` elements.
  ```javascript
  // drops.html:273
  let html = '<table class="dr-table"><thead><tr><th>#</th><th>Player</th>...'
  ```
- **Problem**: Screen readers cannot associate data cells with their column headers.
- **Positive**: The main screener table in lab.js DOES use `scope="col"` (lab.js:1538). This pattern should be replicated.
- **WCAG**: 1.3.1 (Info and Relationships)
- **Recommendation**: Add `scope="col"` to all `<th>` elements in panel tables. Add `<caption>` with the panel name (can be sr-only).

### 17. [MEDIUM] Command palette lacks focus trap -- WCAG 2.4.3
- **File**: `frontend/app.js:1007-1095`
- **Evidence**: The command palette opens with `openCmdPalette()` and focuses the input field. It has Escape key handling and arrow key navigation. However, there is no focus trap -- Tab can move focus to elements behind the palette backdrop.
- **WCAG**: 2.4.3 (Focus Order)
- **Recommendation**: Add Tab/Shift+Tab trapping similar to the auth modal implementation.

### 18. [MEDIUM] Command palette lacks dialog role -- WCAG 4.1.2
- **File**: `frontend/app.js:1017-1029`
- **Evidence**: The command palette is built as:
  ```javascript
  var html = '<div class="cmd-palette-backdrop" id="cmdPalette">' +
    '<div class="cmd-palette">' + ...
  ```
  No `role="dialog"`, `aria-modal`, or `aria-label` is set.
- **WCAG**: 4.1.2 (Name, Role, Value)
- **Recommendation**: Add `role="dialog"`, `aria-modal="true"`, and `aria-label="Quick search"` to the palette container.

### 19. [MEDIUM] Form inputs on panel pages lack labels -- WCAG 1.3.1, 3.3.2
- **Files**: Various panel pages
- **Evidence**: Zero `<label>` elements found across all 74 HTML files. Select elements and search inputs on panel pages rely on visual proximity or placeholder text.
  - `frontend/league-intel.html:63` -- `<input>` in connect form with no label
  - Panel page controls (season selectors, team selectors, search inputs) generally have visual labels nearby but no programmatic association.
- **Positive**: Several panel pages DO use `aria-label` on their controls (advantage.html:90 has `aria-label="Season"`). But this is inconsistent.
- **WCAG**: 1.3.1, 3.3.2
- **Recommendation**: Add `aria-label` to all form controls that lack `<label>` elements. Ideally, use visible `<label>` elements with `for` attributes.

### 20. [MEDIUM] Error messages not associated with form fields -- WCAG 3.3.1
- **File**: `frontend/app.js:543,550`
- **Evidence**: Auth error messages are in `<div>` elements with IDs (`authLoginError`, `authRegisterError`) but are not linked to form inputs via `aria-describedby`. The error text is injected dynamically but screen readers may not announce it.
- **WCAG**: 3.3.1 (Error Identification)
- **Recommendation**: Add `aria-describedby="authLoginError"` to the relevant form inputs. Add `role="alert"` to the error `<div>` so changes are announced.

### 21. [MEDIUM] Hamburger menu panel lacks role and label -- WCAG 4.1.2
- **File**: `frontend/app.js:67`
- **Evidence**: The mobile navigation panel is created as a plain `<div>` with class `mobile-nav-panel`. It has no `role="dialog"` or `role="navigation"`, and no `aria-label`.
- **Positive**: The hamburger button has `aria-label="Open navigation menu"` and `aria-expanded` state management. Escape key closes the panel.
- **WCAG**: 4.1.2 (Name, Role, Value)
- **Recommendation**: Add `role="navigation"` and `aria-label="Mobile navigation"` to the panel.

### 22. [LOW] Touch targets below 44x44px on mobile -- WCAG 2.5.5
- **File**: `frontend/styles.css:840-848`
- **Evidence**: At 768px breakpoint, buttons shrink:
  ```css
  .btn-chunky, .btn-primary { font-size: 11px; padding: 5px 10px; }
  .chip { font-size: 10px; padding: 2px 7px; }
  ```
  At 375px:
  ```css
  .btn-chunky, .btn-primary { font-size: 10px; padding: 4px 7px; }
  .chip { font-size: 9px; padding: 4px 8px; }
  ```
  Chips at 375px are likely well below 44x44px.
- **Positive**: Mobile nav links have `min-height: 44px` (styles.css:269). The theme/signin buttons also have `min-height: 44px`.
- **WCAG**: 2.5.5 (Target Size, Level AAA) / 2.5.8 (Target Size Minimum, Level AA in WCAG 2.2)
- **Recommendation**: Ensure all interactive elements maintain at least 24x24px minimum (WCAG 2.2 AA) or ideally 44x44px on mobile.

### 23. [LOW] Tag picker and note editor popups lack dialog semantics -- WCAG 4.1.2
- **Files**: `frontend/styles.css:1206-1261` (tag picker), `frontend/styles.css:1299-1361` (note editor)
- **Evidence**: These are positioned fixed popups with no `role`, `aria-label`, or focus management.
- **Recommendation**: Add `role="dialog"` and `aria-label` when opened, trap focus, restore focus on close.

### 24. [LOW] Toast notifications not announced to screen readers -- WCAG 4.1.3
- **File**: `frontend/app.js:347-361`
- **Evidence**: Toasts are injected as plain `<div>` elements with no `role="status"` or `aria-live`.
- **Recommendation**: Add `role="status"` and `aria-live="polite"` to the toast container.

### 25. [LOW] prefers-reduced-motion only in styles.css -- Partial
- **File**: `frontend/styles.css:1426-1433`
- **Evidence**: The `prefers-reduced-motion` media query correctly disables CSS animations. However, JavaScript-driven animations (hover effects, command palette opening, toast sliding) are not affected.
- **Recommendation**: Check `window.matchMedia('(prefers-reduced-motion: reduce)')` in JS for any animated transitions.

## WCAG 2.1 Level AA Conformance: DOES NOT CONFORM

The primary remaining barriers are:
1. Color contrast on `--ink-light` text (3.3:1 vs required 4.5:1)
2. Missing visible focus indicators on ~12 input/textarea elements
3. 19 lab dialogs without focus trapping or aria-labelledby
4. No form labels (only placeholders) on auth inputs
5. No aria-live on dynamic data regions across 60+ panel pages

## Trend from Prior Review

Substantial progress. The 2026-03-13 review listed 5 CRITICAL accessibility issues. All 5 have been addressed:
- maximum-scale=1.0 removed from all 74 pages
- nav-search-hint changed from span to button
- role="dialog" added to all modals
- outline:none partially addressed (auth form has focus-visible now)
- aria-label added to all nav elements

The remaining issues are MEDIUM severity and represent the next tier of accessibility work. The codebase has moved from "would fail an automated audit on critical items" to "passes basic automated checks but has manual testing gaps."
